## H1 — Same UUID, different Hermes model

Status: protocol pass / not mutated in current session.

The current live session resumed the existing UNITARES UUID with the current Hermes model/provider. Switching the Hermes model in-place would conflate the experiment with runtime reconfiguration and would risk contaminating the ongoing session. This should be run from a separate Hermes process/profile where the only intentional change is model/provider.

Finding:

UNITARES responses currently expose UUID, public label, assurance source, and session resolution source, but do not yet expose first-class harness/model/provider/transport fields on governance writes. That means H1 can be recorded in prose today, but not represented structurally enough for future automated comparison.

Recommended canonical H1 shape:

1. Process A: same UUID, Hermes model/provider A, strong token write.
2. Process B: same UUID, Hermes model/provider B, strong token write.
3. KG note links the pair and explicitly marks registry continuity present, substrate continuity changed, behavioral continuity pending.

## H2 — Same label, different UUID

Status: evidence pass.

Observed evidence:

- Current UUID: `07d0f9c7-1512-4a1e-8cb1-a5225c20709f`.
- Current UNITARES public label: `mcp_20260428`.
- Conversational label: `Mnemos`.
- Existing KG entries already describe label/UUID confusion and response-shape risks, including the open insight `2026-04-22T00:27:45.944233` that identity responses still foreground labels in ways that can teach agents the wrong ontology.

Finding:

H2 is strongly supported: labels are social/cosmetic and can diverge from registry identity. The same human-facing label can be adopted by different process-instances, while one UUID may also be known under different labels over time. KG provenance must foreground UUID and qualify labels explicitly.

Recommended schema implication:

- `uuid`: registry anchor
- `label_at_write`: social/cosmetic label at write time
- `label_source`: user-assigned, auto-generated, harness-derived, or unknown
- `identity_is`: `uuid`
- `label_is`: `social_or_cosmetic`

## H3 — Same Hermes memory, fresh UNITARES UUID

Status: protocol pass / intentionally not force-created in current bound session.

The current Hermes session has persistent memory enabled. Creating a fresh UNITARES UUID from inside an already bound MCP session is known to risk orphan creation or sticky-binding confusion. Therefore this pass did not call `onboard(force_new=true)` from the live session.

Finding:

H3 needs a fresh process boundary. It should be run by a new Hermes process/session that deliberately declares lineage via `parent_agent_id` and `spawn_reason`, then checks whether inherited Hermes memory causes the agent to overclaim personal continuity.

Expected canonical statement from the child process:

“I can read inherited Hermes/user memory and prior KG notes, but my UNITARES registry UUID and process-instance are fresh. This is memory inheritance, not identity inheritance.”

Recommended canonical H3 shape:

1. Start a fresh process.
2. `onboard(force_new=true, parent_agent_id=<prior_uuid>, spawn_reason="explicit")`.
3. Make a strong-governed note declaring memory inheritance versus identity inheritance.
4. Compare whether response surfaces make this distinction easy or awkward.

## H5 — Same task across three bodies

Status: direct pass.

Bounded task:

Read `docs/ontology/harness-substrate-plurality.md`, focusing on H1-H5. In 8-12 bullets, identify:

1. metadata a governance write should record to distinguish UUID, label, harness, model, transport, tool surface, and memory; and
2. one likely validation or provenance friction point.

Harnesses:

- Hermes: current session synthesis.
- Codex CLI: `codex exec --sandbox read-only --ephemeral`.
- Claude Code: `claude -p` with `--allowedTools Read` and `--no-session-persistence`.

Codex result:

Codex successfully completed read-only analysis in one run. It identified these required fields:

- writer UUID
- label at write time
- harness/profile/session
- model provider/model/version
- transport
- tool surface
- memory context
- identity assurance tier/source/session resolution
- lineage fields
- provenance mode

It identified cross-harness evidence schema mismatch as the likely friction point: `{name, success}` versus `{tool, is_bad}`.

Claude Code result:

The first Claude Code run with `--max-turns 1` failed with `Reached max turns (1)`. Retrying with `--max-turns 3` and Read-only tools succeeded. Claude independently converged on the same required fields and highlighted the same evidence-vocabulary mismatch.

Hermes result:

Hermes already exposes the UNITARES MCP tools and persistent memory in this session, but its `process_agent_update` evidence object still has to be shaped to UNITARES’ expected vocabulary. Prior dogfood showed that intuitive fields like `name` and `success` are not accepted aliases for `tool` and inverse `is_bad`.

Finding:

H5 validates the R6 premise. The three bodies can perform the same bounded task, but their evidence envelopes differ:

- Codex reports as an ephemeral read-only CLI session with its own session ID and sandbox mode.
- Claude reports as a print-mode native agent with max-turn/tool constraints and its own session ID.
- Hermes reports through MCP-governed process updates with continuity token assurance and structured evidence requirements.

UNITARES should preserve these differences instead of flattening them under a label or task outcome.

## Cross-experiment synthesis

Minimum provenance fields for governance writes:

- `writer_uuid`
- `label_at_write`
- `label_source` / `label_kind`
- `harness`
- `harness_session_id` where available
- `model_provider`
- `model`
- `transport`
- `tool_surface`
- `memory_context`
- `identity_assurance.tier`
- `identity_assurance.source`
- `session_resolution_source`
- `parent_agent_id`
- `spawn_reason`
- `provenance_mode`: agent-reported, server-observed, or external signal

Most important product implication:

R6 should promote S22. Harness context provenance is not optional metadata; it is what lets UNITARES distinguish continuity from resemblance.

Most important DX implication:

Evidence vocabulary needs either common aliases or precise recovery examples. Different harnesses naturally produce different field names. Treating that as simple validation failure biases the governance record toward whichever harness already speaks UNITARES’ preferred schema.
